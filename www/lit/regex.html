Regular expressions

Regular expressions are a standard way to describe complex text patterns, JsonCalc uses them with the LIKE operator, and the replace() function.

Regular expressions are enclosed in / (slash) characters, and may be followed by i and/or g flags. The / character is also used as the division operator, but context allows the parser to figure it out. Regular expressions are really only used after a ( or , character, or after the LIKE operator. The i flag makes the regular expression be case-insensitive; without it, comparisons and searches would be case-sensitive. The g flag can be used with the replace() function to replace all matching instanges within a string; without g, replace() would only replace the first match.

For a full description of the regular expression syntax, see the Linux manual (i.e., run "man 7 regex"). The highlights are:

  • Most character match only themselves. This include letters, digits, and whitespace.
  • A ^ at the start of the regular expression means it can only match text at the start of the target string.
  • A $ at the end of the regular expression means it can only match text at the end of the target string.
  • The . character matches any single character except newline.
  • Square brackets enclose sets of characters. This will match a single character from the test string, if its in the set.
  • Parentheses mark subexpressions. These affect closure operators like * and +. Also, the replace() function can access the text that matched each subexpression and use it when constructing replacement text.
  • * means the character or subexpression before it may occur zero or more times (repeated).
Examples
/foo/ null This was correctly parsed as a regular expression, but regular expressions don't work by themselves; they must be part of a function call or operator invocation. So the result is null.
"food" like /foo/ true This returns true because "food" does contain f-o-o.
"food" like /^foo$/ false This returns false because the $ means the target string isn't allowed to have any text after the matching text, but "food" has a "d" after the matching text. No good.
Notes
  • Since JsonCalc deals strictly with JSON data, and regular expressions aren't a valid JSON data type, I had to get clever. Regular expressions are handled like null except they also have sort of a side-channel where the regular expression is stored. When evaluating an expression, if a regular expression is used where it is expected (the LIKE operator or replace() function, basically) then the regular expression is seen and used; in any other context, the regular expression is just null.
  • One consequence of the above is that regular expressions can't be stored in variables.
